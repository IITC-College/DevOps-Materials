Welcome to Exercise 3!

CRITICAL SECURITY ISSUE: The engineering team's API keys file has permissions
666 (rw-rw-rw-). This means ANYONE on the system can read AND modify the
API keys. This is a serious security vulnerability!

BACKGROUND:
-----------
The `stat` command shows detailed file information, including NUMERIC
permissions (which ls -l doesn't show directly):

  stat company/engineering/api_keys.env

Look for the "Access" line - it shows permissions in both formats:
  Access: (0666/-rw-rw-rw-)

EXERCISE:
---------
Secure the API keys file AND verify with stat. Then also fix the employee
reviews file using what you've learned.

INSTRUCTIONS:
-------------
1. Check the current permissions with BOTH ls -l and stat:

   ls -l company/engineering/api_keys.env
   stat company/engineering/api_keys.env

2. See the problem: 666 means everyone can read AND write this file.
   This is a security nightmare for API keys!

3. Fix it - API keys should ONLY be readable by the owner (not even writable
   by group or others). Use NUMERIC mode:

   sudo chmod 600 company/engineering/api_keys.env

4. Verify with stat:

   stat company/engineering/api_keys.env

   You should see: Access: (0600/-rw-------)

5. Now fix the employee reviews file. It's also too open.
   Check it first:

   ls -l company/hr/employee_reviews.txt

6. Fix it using SYMBOLIC mode this time - remove read for group and others:

   sudo chmod go-r company/hr/employee_reviews.txt

7. Verify both fixes:

   ls -l company/engineering/api_keys.env
   ls -l company/hr/employee_reviews.txt

COMMANDS TO USE:
----------------
- ls -l company/engineering/api_keys.env
- stat company/engineering/api_keys.env
- sudo chmod 600 company/engineering/api_keys.env
- stat company/engineering/api_keys.env
- ls -l company/hr/employee_reviews.txt
- sudo chmod go-r company/hr/employee_reviews.txt
- ls -l company/hr/employee_reviews.txt

KEY CONCEPT - PRINCIPLE OF LEAST PRIVILEGE:
--------------------------------------------
Always give the MINIMUM permissions needed:
  - API keys → 600 (only owner reads/writes)
  - Scripts  → 750 (owner full, group can run, others nothing)
  - Public docs → 644 (owner writes, everyone reads)
  - Shared files → 660 (owner and group read/write, others nothing)

NEVER use 777 or 666 in production. If you see these permissions, they are
almost certainly WRONG.
