        "The Leaky Reports" - Securing Confidential Data

Welcome to Exercise 2!

The security audit flagged TWO report files with dangerously open permissions.
The weekly summary has 666 (world-writable!) and the security audit report
has 777 (FULL access for everyone!). Both are owned by root:root.

This exercise tests your ability to analyze a situation and choose the RIGHT
permissions for different types of data.

BACKGROUND:
-----------
Different types of data need different permission levels:

  Public reports    → 644 (rw-r--r--) owner writes, everyone reads
  Team reports      → 640 (rw-r-----) owner writes, team reads
  Confidential      → 600 (rw-------) owner only
  Executable report → 750 (rwxr-x---) owner full, team reads/runs

The KEY QUESTION: "Who NEEDS access to this file?"
  - If everyone: 644
  - If just the team: 640 or 660
  - If just one person: 600

EXERCISE:
---------
Fix both report files with appropriate ownership and permissions.

INSTRUCTIONS:
-------------
1. Check the current state:

   ls -l data/reports/

2. Analyze each file:

   weekly_summary.txt (666, root:root):
   → This is a GENERAL report - everyone should READ it, but only the
     manager should WRITE it.
   → Correct: owner=manager, group=engineering, permissions=644

   security_audit.txt (777, root:root):
   → This is a CONFIDENTIAL report - only the manager should have access.
   → Correct: owner=manager, group=devops, permissions=600

3. Fix weekly_summary.txt:

   sudo chown manager:engineering data/reports/weekly_summary.txt
   sudo chmod 644 data/reports/weekly_summary.txt

4. Fix security_audit.txt:

   sudo chown manager:devops data/reports/security_audit.txt
   sudo chmod 600 data/reports/security_audit.txt

5. Verify:

   ls -l data/reports/

6. Test access - try reading both files:

   cat data/reports/weekly_summary.txt

   cat data/reports/security_audit.txt

   The weekly summary should be readable (644 = others can read).
   The security audit should show "Permission denied" (600 = owner only).

COMMANDS TO USE:
----------------
- ls -l data/reports/
- sudo chown manager:engineering data/reports/weekly_summary.txt
- sudo chmod 644 data/reports/weekly_summary.txt
- sudo chown manager:devops data/reports/security_audit.txt
- sudo chmod 600 data/reports/security_audit.txt
- ls -l data/reports/
- cat data/reports/weekly_summary.txt
- cat data/reports/security_audit.txt

KEY CONCEPT - WHY 777 IS DANGEROUS:
-------------------------------------
Permission 777 means:
  - ANYONE can read the file (data leak)
  - ANYONE can modify the file (data tampering)
  - ANYONE can execute the file (if it's a script, anyone can run it)

A security audit report with 777 means:
  - Any user can read sensitive security findings
  - Any user can MODIFY the report (change the findings!)
  - This completely defeats the purpose of a security audit

Rule of thumb: If you ever see 777 or 666 on a file, it's almost
certainly a security problem that needs to be fixed.
