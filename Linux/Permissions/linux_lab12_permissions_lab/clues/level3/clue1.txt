        "Broken Automation" - Fixing Scripts (Ownership + Permissions)
================================================================================

Welcome to Level 3!

The automation scripts in the scripts/ directory are completely broken:
- They're all owned by root:root (wrong owner AND wrong group)
- None of them have execute permission (can't be run)
- Each script needs DIFFERENT ownership depending on who should run it

This is a real-world scenario: you need to fix BOTH ownership AND permissions.

BACKGROUND:
-----------
Script permissions follow a common pattern:
  - Owner: the user responsible for running/maintaining the script
  - Group: the team that should also be able to run it
  - Execute: MUST be set for the script to run
  - Write: only for the owner (don't let others modify scripts)

Common script permissions:
  750 → Owner: rwx, Group: r-x, Others: --- (team script)
  700 → Owner: rwx, Group: ---, Others: --- (private script)
  755 → Owner: rwx, Group: r-x, Others: r-x (public script)

EXERCISE:
---------
Fix all three scripts with the correct ownership AND permissions.

INSTRUCTIONS:
-------------
1. Check the current state of all scripts:

   ls -l scripts/

2. All three are owned by root:root with 644. They need:

   backup_database.sh:
   → Owner: developer  Group: engineering  Permissions: 750
   (Developer maintains it, engineering team can run it, others can't)

   generate_report.sh:
   → Owner: manager  Group: devops  Permissions: 750
   (Manager maintains it, DevOps team can run it, others can't)

   cleanup_logs.sh:
   → Owner: manager  Group: devops  Permissions: 700
   (Manager only - this is a dangerous script, no group access)

3. Fix backup_database.sh:

   sudo chown developer:engineering scripts/backup_database.sh
   sudo chmod 750 scripts/backup_database.sh

4. Fix generate_report.sh:

   sudo chown manager:devops scripts/generate_report.sh
   sudo chmod 750 scripts/generate_report.sh

5. Fix cleanup_logs.sh (notice: 700, not 750 - private to manager):

   sudo chown manager:devops scripts/cleanup_logs.sh
   sudo chmod 700 scripts/cleanup_logs.sh

6. Verify ALL fixes:

   ls -l scripts/

7. Test that backup_database.sh is now executable:

   ./scripts/backup_database.sh

   (This may fail if you're not the owner or in the group. That's OK!
   The point is the permissions are now CORRECT.)

COMMANDS TO USE:
----------------
- ls -l scripts/
- sudo chown developer:engineering scripts/backup_database.sh
- sudo chmod 750 scripts/backup_database.sh
- sudo chown manager:devops scripts/generate_report.sh
- sudo chmod 750 scripts/generate_report.sh
- sudo chown manager:devops scripts/cleanup_logs.sh
- sudo chmod 700 scripts/cleanup_logs.sh
- ls -l scripts/

KEY CONCEPT - 750 vs 700:
--------------------------
750 (rwxr-x---):
  - Owner can read, write, execute
  - Group can read and execute (but NOT modify)
  - Others have no access
  → Use when a TEAM needs to run the script

700 (rwx------):
  - Owner can read, write, execute
  - Nobody else has any access
  → Use for SENSITIVE scripts (destructive operations, admin tasks)
