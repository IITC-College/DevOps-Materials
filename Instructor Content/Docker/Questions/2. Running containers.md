## Question 1

**שאלה:**
A developer runs a container using `docker run -d nginx`. The command returns a container ID and frees up their terminal. Later, they want to view the real-time access logs generated by the Nginx web server inside the container. Which of the following commands is the correct and most effective way to do this?

**אפשרויות:**

1. `docker ps --logs <container_name_or_id>`
2. `docker exec <container_name_or_id> cat /var/log/nginx/access.log`
3. `docker run -it nginx logs`
4. ✅ `docker logs -f <container_name_or_id>`
5. `docker attach <container_name_or_id>`

**תשובה נכונה:** אפשרות 4
(`-f` / `--follow` מציג לוגים בזמן אמת)

## Question 2

**שאלה:**
A developer runs `docker run ubuntu echo "hello"`. The command prints "hello" and immediately exits. A few seconds later, they run `docker ps` and see no running containers. Why did the container stop?

**אפשרויות:**

1. An error occurred inside the Ubuntu container, causing it to crash.
2. The `-d` (detached) flag was missing, so the container stopped as soon as it was launched.
3. ✅ The container's main process (`echo "hello"`) completed its task and exited successfully. Since the container had no other long-running process to keep it alive, Docker stopped the container.
4. The docker run command only creates a container; you must use docker start to keep it running.
5. Ubuntu images are not designed to be run as containers and will always stop immediately.

**תשובה נכונה:** אפשרות 3
(חיי container קשורים לתהליך הראשי שלו)

## Question 3

**שאלה:**
A developer needs to stop a running container gracefully, giving its main process time to shut down and save its state. However, if it doesn't stop within a few seconds, they want it to be terminated forcefully. Which command should they use?

**אפשרויות:**

1. ✅ `docker stop <container_name_or_id>`
2. `docker kill <container_name_or_id>`
3. `docker pause <container_name_or_id>`
4. `docker rm <container_name_or_id>`
5. `docker exec <container_name_or_id> shutdown`

**תשובה נכונה:** אפשרות 1

`docker stop` שולח SIGTERM ומחכה (ברירת מחדל ~10 שניות), ואם לא נסגר – שולח SIGKILL.

## Question 4

**שאלה:**
A developer runs `docker ps -a` ורואה את הפלט הבא:

* `web1` → Status: Up 2 minutes
* `task1` → Status: Exited (0) 8 minutes ago

מה ההבדל המרכזי בין שני הקונטיינרים?

**אפשרויות:**

1. Both containers are running, but task1 is running in detached mode.
2. web1 has crashed, and task1 is waiting to be started.
3. web1 is a container, but task1 is an image.
4. web1 was started with docker run, while task1 was started with docker start.
5. ✅ web1 is a long-lived container currently running, while task1 was a short-lived container that completed its task and is now stopped.

**תשובה נכונה:** אפשרות 5

`Up` = רץ כרגע
`Exited (0)` = הסתיים בהצלחה

## Question 5

**שאלה:**
A developer has a running container named `my-server`. They want to open an interactive shell inside this container. They run:
`docker run -it my-server sh`
and it fails. What is the mistake?

**אפשרויות:**

1. The container name my-server is invalid; only the container ID can be used.
2. The `-it` flags are incorrect; they should be `-i -t`.
3. The shell command must be an absolute path, like `/bin/sh`.
4. ✅ The developer should have used `docker exec`, not `docker run`.
5. An interactive shell cannot be opened in a container that was started in detached (-d) mode.

**תשובה נכונה:** אפשרות 4

הפקודה הנכונה:

```
docker exec -it my-server sh
```

`docker run` יוצר container חדש — לא נכנס לקיים.

## Question 6

**שאלה:**
A developer has created a Dockerfile in their current directory. The build process requires files from the current directory to be copied into the image. Which command correctly builds the image and tags it as `my-app:v1`?

**אפשרויות:**

1. ✅ `docker build -t my-app:v1 .`
2. `docker run -t my-app:v1 .`
3. `docker build Dockerfile -t my-app:v1`
4. `docker image create my-app:v1 .`
5. `docker build -f . -t my-app:v1`

**תשובה נכונה:** אפשרות 1

הנקודה (`.`) מגדירה את ה-build context (התיקייה הנוכחית).
